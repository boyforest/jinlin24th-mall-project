## 金霖二十四养 — 管理后台体验分析与优化建议

> 分析视角：产品经理（关注信息架构、操作效率、认知负担）
> 分析对象：admin-web 前端代码（React + Ant Design 6）

---

### 一、核心问题诊断

#### 1. 导航结构扁平，缺乏业务分组

当前侧边栏一次性平铺了 14 个菜单项，没有任何层级关系。对运营人员来说，"商品管理"和"库存流水"虽然都跟货有关，但混在"客户管理""分销管理"中间，找起来全靠记忆。

**影响**：新员工上手慢，日常操作需要来回扫视菜单。

**建议重构为业务域分组**：

| 分组 | 包含模块 |
|------|----------|
| 概览 | 首页 Dashboard |
| 商品中心 | 商品管理、规格管理、分类管理 |
| 交易中心 | 订单管理、优惠券、活动运营 |
| 仓储物流 | 仓库管理、库存管理、库存流水 |
| 客户与分销 | 客户管理、跟进记录、分销管理 |
| 系统设置 | 用户管理（管理员账号） |

分组后菜单从"14 选 1"降为"先选域（5 组），再选页面"，认知复杂度显著下降。

---

#### 2. 所有页面长一个样——"万能 CRUD 表格"模式

几乎所有页面都是同一个 `CrudTable` 组件的不同参数配置：筛选栏 + 表格 + 新增/编辑弹窗。这种做法对开发效率友好，但对用户体验有两大伤害：

- **缺乏业务语义**：订单管理和客户管理的操作逻辑完全不同，但界面上看不出区别。用户无法通过视觉快速辨别"我在哪"。
- **信息密度失衡**：有些页面列太多（分销管理有 12 列），横向滚动严重；有些页面列太少但表格依然全屏铺开。

**建议**：

- 为高频业务页面（订单、商品、客户）设计专属布局，加入统计卡片、状态看板等辅助视图。
- CrudTable 保留给低频、配置类页面（仓库管理、规格管理等），降低维护成本的同时不牺牲核心页面体验。

---

#### 3. ID 驱动的关联关系，用户被迫当"人肉数据库"

这是当前最影响可用性的问题。多处字段要求输入或展示原始 ID：

| 页面 | 问题字段 | 用户实际需要的 |
|------|----------|----------------|
| 库存管理 | 仓库ID、SKU ID | 仓库名称 + SKU名称（可搜索下拉） |
| 跟进记录 | 客户ID、跟进人ID | 客户名称 + 员工姓名（可搜索下拉） |
| 客户管理 | 销售ID | 销售员姓名 |
| 订单管理 | 用户ID、推荐官 | 用户昵称/手机号 |

**影响**：运营人员必须先去别的页面查 ID、记住、再回来填写。这种"上下文切换"是体验差的最大根源。

**建议**：

- 表格列：ID 替换为名称（tooltip 显示 ID）
- 表单字段：使用带搜索的 Select 或 AsyncSelect 组件，输入关键字远程匹配，展示名称、存储 ID
- 筛选器同理：用名称搜索代替手动输入 ID

---

#### 4. Dashboard 形同虚设

首页只展示了 7 个静态数字（总量统计），没有趋势、没有对比、没有待办提醒。管理后台的首页应该是"今天我该关注什么"的入口。

**建议优化方向**：

- **今日关键指标**：今日新增订单数、今日营收、待发货订单、库存预警商品数
- **趋势图**：最近 7 天/30 天的订单量、收入折线图
- **待办事项**：待发货订单 N 笔、待结算佣金 N 条、库存告警 N 个 SKU
- **快捷入口**：一键跳转到待处理任务

---

#### 5. 详情页展示原始 JSON

`CrudTable` 的"查看"功能弹窗直接 `JSON.stringify` 展示数据：

```jsx
<pre className="detail-json">{JSON.stringify(detail, null, 2)}</pre>
```

运营人员看到的是 `"categoryId": 3, "status": 1` 这样的原始数据，完全不可读。

**建议**：

- 详情页用 Descriptions（Ant Design 描述列表）组件展示，字段有中文标签
- 枚举值显示为可读文本（如 status 1 → "启用"）
- 图片字段直接预览而非显示 URL 字符串
- 关联字段展示名称而非 ID

---

#### 6. 时间输入要求手写 ISO 格式

"活动运营"和"跟进记录"中的时间字段是普通 Input，placeholder 示例为 `2026-05-20T10:00:00`。

这对运营人员来说极不友好，手动输入极易出错。

**建议**：使用 Ant Design 的 `DatePicker` 或 `DateTimePicker`，带直观的日历弹窗选择。

---

#### 7. 缺乏实体间的导航关联

当前各页面是完全孤立的。举例：

- 在"客户管理"看到一个客户，想看他的跟进记录 → 需要记住客户 ID，切到"跟进记录"页，手动输入筛选
- 在"订单管理"看到一笔订单，想看这个商品 → 没有入口
- 在"库存管理"发现库存告警，想看是哪个仓库 → 只有仓库 ID

**建议**：

- 表格中的关联字段做成可点击链接（如客户名称 → 跳转客户详情）
- 客户详情页内嵌跟进记录、订单记录 Tab
- 商品详情页内嵌 SKU 列表、库存情况
- 订单详情页展示商品信息、收货信息、物流状态时间线

---

#### 8. 缺少批量操作

常见场景：

- 批量发货（选中多个待发货订单一键发货）
- 批量结算佣金
- 批量调整商品上下架状态
- 批量导入/导出客户

当前每笔操作都需要逐条点击确认，效率低下。

**建议**：在 Table 上启用 `rowSelection`，工具栏增加批量操作按钮。

---

### 二、优化优先级建议（按 ROI 排序）

| 优先级 | 改进项 | 预期效果 | 实现难度 |
|--------|--------|----------|----------|
| P0 | ID 替换为名称 + 搜索选择器 | 消除最大痛点，减少 60%+ 的上下文切换 | 中 |
| P0 | 时间字段改用 DatePicker | 几乎零成本消除输入错误 | 低 |
| P1 | 导航菜单分组 | 降低认知负担，新人上手更快 | 低 |
| P1 | 详情弹窗格式化展示 | 让"查看"功能真正可用 | 中 |
| P1 | Dashboard 增加待办和趋势 | 首页从摆设变为决策入口 | 中 |
| P2 | 核心页面专属布局 | 订单/客户页面效率提升 | 高 |
| P2 | 实体间关联导航 | 减少页面跳转，形成工作流 | 中 |
| P3 | 批量操作 | 高频运营场景效率翻倍 | 中 |

---

### 三、总结

你的直觉是对的——当前系统的核心问题在于：**它是"面向数据库表结构"设计的，而不是"面向运营工作流"设计的**。每个页面对应一张数据库表，字段直接暴露给用户，关联关系只有裸 ID。

优化的核心思路是：**让系统适应人的工作方式，而不是让人适应系统的数据结构**。

具体来说就是三件事：
1. 把 ID 藏起来，让用户看到和操作的永远是"名称"
2. 把孤立的表连起来，让用户能在一个上下文里完成一个完整任务
3. 区分高频和低频场景，高频场景值得投入专属设计
